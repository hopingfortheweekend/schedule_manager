> 文档目标：提炼开发过程中的思考与收获，用于面试交流和简历素材。

## 1. 项目概述

| 项目 | 说明 |
|------|------|
| 名称 | 轻简日程管理 (Light Schedule Manager) |
| 类型 | Python Tkinter 桌面应用 |
| 周期 | 2026 年 6 月，约 2 周（15 个 Git 提交） |
| 角色 | 独立全流程（产品规划 → 需求文档 → 架构设计 → 开发 → 测试 → 打包分发） |
| 仓库 | https://github.com/hopingfortheweekend/light_schedule |

### 一句话定位

面向有拖延倾向的大学生/职场新人的**本地化日程管理工具**，核心差异点是：日历上按颜色区分 DDL 紧急程度，让用户一眼看清"我还有多少时间"。

---

## 2. 做这件事的初衷

市面上日程管理软件很多（滴答清单、Todoist、Notion），但它们要么功能太重、要么需要联网、要么操作不够直观。我希望做一个：
- 打开即用的本地工具，不需要注册登录
- 能一眼看到 DDL 颜色分布（比纯文字列表更有时间紧迫感）
- 把大项目拆成小步骤，逐个击破

这也是一个实践项目——我想通过它证明自己具备**从需求到落地的完整闭环能力**。

## 4. 产品思维体现

### 4.1 需求优先级判断

在写 PRD 时，我用 MoSCoW 方法把 15 项功能分了四个优先级，明确**这版做什么、不做什么**，防止范围蔓延：

- **Must（3 项）**：删除功能、统计看板、DDL 当日汇总 — 没有就不算完成
- **Should（1 项）**：CSV 导出 — 重要但非紧急
- **Could（1 项）**：视觉优化 — 后续版本考虑
- **Won't（3 项）**：用户明确砍掉（未完成原因记录、多用户同步、移动端）

> **收获**：使用这种优先度排列原则有助于理清产品功能上新时的思路，明确哪些是本次更新必须包括的、哪些是重要但不急于这一次改成的、哪些是可以考虑的以及哪里是产品目前的边界，有助于我们从功能上对产品定位和目标有一个整体的认知。

### 4.2 迭代中的用户反馈

一些改动来自实际使用体验，而非 PRD 预设：

| 反馈 | 改动 |
|------|------|
| "项目详情窗口太小，每次打开都要拖" | 默认尺寸从迷你改为 480×380 |
| "删除按钮和编辑混在一起容易误操作" | 删除移到列表底部独立按钮，编辑弹窗去掉删除 |
| "导出按钮独占一列很突兀" | 改为叠放在标签栏右侧，跟页面标签同行 |
| "点击状态框有时操作的是错的条目" | 排查出排序后索引映射 bug，重构为 `_display_items` 追踪 |
| "步骤取消了项目还是已完成的" | 补充 auto_check_project 的反向逻辑 |

> **收获**：PRD 只是起点，真正打磨体验靠自测和使用反馈。很多问题不在规划范围内，但会切实影响用户感受。产品的最终目的是服务用户，如果可以集思广益，其改进会在质与量上有较大的提升，同时对用户、企业来说也是好事。

## 5. 技术改进

### 5.1 索引映射问题

**场景**：长期项目的步骤按截止日期排序显示，但操作（切换状态、编辑内容）需要操作实际数据位置。

**问题**：排序后 visual index ≠ data index，点击状态框操作了错误的条目。

**方案**：引入 `_display_to_data` 映射表，在显示层和存储层之间做一层转换：
```
显示第 0 条 → _display_to_data[0] = 3 → 操作 data[3]
```

**一些启发**：排序和过滤场景下，永远不要在排序后的列表上用 index 直接操作源数据。

### 5.2 数据流设计

```
用户操作 → UI 层（task_tab / project_tab）
         → 数据层（DataStore CRUD）
         → JSON 文件持久化
         → _on_data_changed 回调
         → 日历视图 + 统计看板自动刷新
```

数据只往一个方向流动（UI → DataStore → File），查询也只通过 DataStore 方法，UI 层不认识 JSON。这让后面加功能时踩坑少了很多。

### 5.3 PyInstaller 打包适配

原代码用 `os.getcwd()` 定位数据文件，打包成 exe 后路径指向临时目录。改为检测 `sys.frozen`，按需使用 `sys.executable`（exe 所在目录），首次运行自动从打包数据复制示例。

## 6. 从中获得的经验

### 做得好的
- 模块化拆分做对了时间点：代码约 400 行时拆的，再晚可能会更麻烦
- PRD 写用户故事和验收标准，而不是功能清单——这让后续开发不容易跑偏
- 单元测试虽然只有 19 个，但覆盖了所有 CRUD 和边界情况（坏索引、空数据、持久化）
- Conventional Commits 让提交历史本身就是更新日志

### 可以改进的
- 没有做自动化测试集成（CI），每次手动跑 pytest
- 日志写入 app.log 但没有日志轮转，文件会持续增长
- 数据统计模块的 matplotlib 中文支持花了不少时间踩坑，应该更早测试多语言场景
- UI 层面还有优化空间（窗口布局自适应、深色主题等）

### 如果继续做
- 搜索/过滤功能
- 提醒/通知（Windows 系统通知）
- 未完成原因记录（原本在 Won't 里）
- 数据可视化做得更丰富（饼图、热力图）
